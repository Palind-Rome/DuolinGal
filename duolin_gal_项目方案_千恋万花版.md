# DuolinGal 项目方案（修订版 / 《千恋万花》验证线）

## 1. 先说结论

这个项目 **可以做**，但更准确地说，它目前适合被定义成：

**面向《千恋万花》Steam 版的本地研究型工具链，用来验证 KiriKiri 作品的英文语音补丁是否能被工程化生成。**

如果把目标写成“很快做成通用 galgame 英语配音平台”，那就过于乐观了。

## 2. ChatGPT 方案里我认可的部分

- 先只做 **KiriKiri / KiriKiri Z**，这是对的。
- 先拿 **《千恋万花》Steam 版** 做样本，是对的。
- 第一版选择 **离线批量生成**，而不是实时 TTS，也是对的。
- **GPT-SoVITS** 作为优先尝试模型，方向合理。

## 3. ChatGPT 方案里被低估的部分

### 3.1 主风险不是 TTS，而是“提取 + 对齐 + 回注”

真正决定项目能不能落地的，不是模型有没有跨语言能力，而是：

1. 你能不能稳定提取语音与脚本。
2. 你能不能把英文文本和具体语音文件对齐。
3. 你能不能把生成结果重新塞回游戏且稳定播放。

### 3.2 SCN / PSB 不是普通小细节

《千恋万花》这类 KiriKiri Z 作品的关键难点，不是读简单的 `.ks` 文本，而是处理 `.scn` / `.psb` 相关中间格式。  

所以“脚本解析”不能被当作中间环节一笔带过，它其实是第一阶段最关键的验证项之一。

### 3.3 Web UI 不该太早开始

在真实样本还没证明“可提取、可对齐、可回注”之前，太早铺 UI 会制造一种“项目快完成了”的错觉。  

更稳的顺序是：

- 先 CLI
- 再本地 API
- 最后 Web UI

## 4. 冷静版项目定位

DuolinGal 第一阶段应当定位为：

**角色化英语听力实验工具链**

而不是：

- 商业级英语配音系统
- 通用多引擎平台
- 一键全自动整作英配工具

## 5. 第一阶段范围

- 只支持《千恋万花》Steam 版
- 只支持 Windows
- 只支持本地运行
- 只做离线批量生成
- 只做 1 到 2 名角色验证
- 不保证整作覆盖

## 6. 三个必须先过的门槛

### 门槛 A：提取与解析

先证明：

- `voice.xp3` 能稳定提取
- `scn.xp3` 能稳定转出可分析中间层
- 至少有 100 条候选语音台词可供人工复核

如果这一步不过，就不要先碰训练。

### 门槛 B：回注与播放

先证明：

- 手工替换 10 条语音后，游戏内可以正常播放
- 快进、读档至少做一次基础验证

如果这一步不过，就不要先谈批量补丁。

### 门槛 C：单角色音色验证

先证明：

- 选 1 名角色
- 清洗 10 到 20 分钟样本
- 生成一批英文样例后，角色感仍然明显可辨

如果这一步不过，就不要先承诺“整作英配”。

## 7. 第一阶段推荐工具链

- XP3 解包/封包：KrkrExtract
- SCN / PSB 处理：FreeMote
- SCN 字符串编辑参考：KrKrZSceneManager
- 覆盖式补丁验证：KirikiriTools
- 文本工具参考：VNTranslationTools
- 音频后处理：FFmpeg
- TTS 优先尝试：GPT-SoVITS

## 8. 第一阶段最合理的开发顺序

1. 目录分析器
2. 工作区初始化
3. 外部工具封装层
4. SCN / PSB 中间层抽取
5. lines.csv 导出
6. 单角色训练与推理验证
7. 小规模补丁验证
8. 本地 UI

DuolinGal 现在最值得做的，不是“把所有想法一次堆满”，而是：

**先用《千恋万花》验证一条真实可跑通的 KiriKiri 英文语音补丁链路。**
