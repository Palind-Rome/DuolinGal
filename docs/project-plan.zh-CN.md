# DuolinGal 项目方案（修订版）

## 1. 项目定位

DuolinGal 的第一阶段不是做“万能 galgame 英语配音平台”，而是做：

**一套面向《千恋万花》Steam 版的本地工具链，用来验证 KiriKiri 作品的英文配音补丁是否可被工程化生成。**

这意味着第一阶段优先级是：

1. 证明提取链路成立。
2. 证明对齐链路成立。
3. 证明回注链路成立。
4. 最后才是模型集成和 UI 打磨。

## 2. 第一阶段范围

- 只支持《千恋万花》Steam 版
- 只支持 Windows
- 只支持本地运行
- 只做离线批量生成
- 只做单角色或少量角色验证
- 不承诺首期覆盖整部作品

## 3. 第一阶段输出物

### CLI / API

- 游戏目录分析
- 项目工作区初始化
- 外部工具状态检测
- 对齐数据结构与 CSV 导出

### 工作区

- `project_manifest.json`
- `directory_snapshot.json`
- 标准子目录结构

### 文档

- 可行性与风险说明
- 阶段门槛
- 第三方工具接入说明

## 4. 里程碑

### 里程碑 0：链路验证

目标：

- 验证样本游戏目录识别
- 选定真实工具链
- 抽出第一批可分析样本

通过标准：

- 能输出稳定 `project_manifest.json`
- 能记录工具缺失和外部依赖状态

### 里程碑 1：提取器外壳

目标：

- 封装 KrkrExtract / FreeMote / FFmpeg 等工具调用
- 记录命令、返回码、输出日志

通过标准：

- 任一步骤失败时有明确错误日志
- 能断点重跑单个阶段

### 里程碑 2：文本-语音对齐

目标：

- 从 SCN / PSB 中抽出候选台词节点
- 导出统一 lines.csv

通过标准：

- 至少 100 条样本可复核
- 误对齐样本可标记 `needs_review`

### 里程碑 3：单角色 TTS 验证

目标：

- 为 1 名角色生成一批英文样例

通过标准：

- 主观上可辨识角色感
- 英文内容可听懂

### 里程碑 4：最小补丁验证

目标：

- 小范围替换游戏语音并回注

通过标准：

- 10 条样本播放正常
- 至少通过一次快进/读档验证

### 里程碑 5：本地 UI

目标：

- 将 CLI/服务层包装为本地可视化工作流

通过标准：

- 用户无需手工拼命令
- 日志可视化
- 失败可重试

## 5. 当前代码与里程碑的对应关系

当前仓库已经落地：

- `src/duolingal/core/analyzer.py`
  - 识别《千恋万花》样本目录
- `src/duolingal/core/workspace.py`
  - 初始化工作区和清单文件
- `src/duolingal/core/tooling.py`
  - 建模第三方工具依赖
- `src/duolingal/core/aligner.py`
  - 提供对齐数据结构与 CSV stub
- `src/duolingal/api/app.py`
  - 提供最小本地 API 入口
- `src/duolingal/cli.py`
  - 提供命令行入口

## 6. 近期任务

### P0

- 封装外部工具命令执行层
- 为提取器补日志与返回码模型
- 加入项目配置文件读取
- 做一批真实资源验证脚本

### P1

- 接入 SCN / PSB 中间层解析
- 补对齐 heuristics
- 导出第一版 `lines.csv`

### P2

- 接 GPT-SoVITS 外部推理
- 做训练样本清洗清单
- 生成 20 条验证音频

### P3

- 研究补丁封包与回注
- 再决定 Web UI 具体技术方案

## 7. 成功标准

DuolinGal 第一阶段成功，不是“整部《千恋万花》已经全部英配完成”，而是：

- 工程链路可以跑通
- 风险被逐个证伪或证实
- 有一条真正能工作的最小样本线
- 后续扩展不是从零开始
